    2026-07-19 — NMPC를 Autoware 전체 스택 + CARLA에 올리기: 문제와 해결

    전날(2026-07-18) 확정한 NMPC 컨트롤러를 Autoware Universe 전체 스택 + CARLA에 꽂아 실제로 주행시키는 것이 목표였다. 결과는 성공: 우리 nmpc_autoware_node가 Autoware의 기본 컨트롤러를 대체해 CARLA 차량을 코너 포함 약 185m 무사고 완주했다(MRM=NORMAL).

    이 문서는 그 과정에서 막혔던 문제들을 증상 → 원인 → 해결 순으로 남긴다. 대부분은 "Autoware를 축소 구성(perception 없이)으로 돌릴 때" 생기는 것들이라 재현 가능성이 높다.

    0. 최종 구성 요약

    CARLA (sync, -quality-level=Low -RenderOffScreen)
  └─ autoware_carla_interface (e2e_simulator.launch.xml simulator_type:=carla, perception:=false)
       ├─ localization: 실제 GNSS/IMU/LiDAR NDT
       ├─ planning: mission → behavior_path → (motion 공백) → [우리 브리지] → validator
       └─ control: controller_node_exe **unload** → 우리 nmpc_autoware_node 가 단독 발행

    핵심 파라미터(launch/autoware_planning_sim.sh nmpc): cruise_speed_mps=8.33(30km/h), a_lat_max=1.5(코너 target ~14km/h), goal_zone_m=12.0, dt=0.05.

    1. [치명적] conda python3.13 오염 — 인터페이스가 통째로 죽음

    증상 - autoware_carla_interface 프로세스가 즉시 die. /clock 미발행, ego 스폰 안 됨, /localization/kinematic_state 영원히 안 나옴. 스택은 떠 있는데 시뮬레이터가 안 붙음. - 로그: ModuleNotFoundError: No module named 'carla' - 부수 증상: map_hash_generator도 No module named 'rclpy._rclpy_pybind11'로 die.

    원인 carla 파이썬 모듈과 ROS Humble의 rclpy는 python3.10 전용으로 설치돼 있는데 (~/.local/lib/python3.10/site-packages/carla, /opt/ros/humble), 셸에 miniconda(python3.13) 가 활성화돼 있으면 python3가 conda 쪽을 가리킨다. ament 실행파일 shebang이 #!/usr/bin/env python3라 그대로 3.13으로 실행되고, 컴파일된 확장모듈(cp310)이라 임포트가 실패한다. PYTHONPATH를 더해도 소용없다 (No module named 'carla.libcarla' — ABI 불일치).

    해결 런치 wrapper에서 시스템 python 우선: 

    export PATH="/usr/bin:$PATH"   # python3 -> /usr/bin/python3 (3.10)
    확인: 런치 로그에 python3 = /usr/bin/python3 (Python 3.10.12)가 찍혀야 한다. 보조 스크립트는 /usr/bin/python3를 명시 실행한다.

    2. [치명적] nohup 누락 — 스택이 SIGKILL로 몰살

    증상 백그라운드로 띄운 e2e 스택이 부팅 도중 ndt_scan_matcher, map_container, rviz2 등이 줄줄이 exit code -9로 die. /clock이 끝내 안 나옴.

    원인 bash ... &로만 띄우면 호출한 셸이 끝날 때 SIGHUP이 프로세스 그룹에 전파되어 스택 전체가 죽는다. (도구가 명령 단위로 셸을 띄웠다 닫는 환경에서 특히 잘 물린다.)

    해결 

    nohup setsid bash wrapper.sh </dev/null >log 2>&1 & disown

    3. rviz2 중복 / 오펀 노드 — duplicated_node_checker ERROR

    증상 - rviz2가 2개, control 노드들이 2개씩 보임. duplicated_node_checker가 ERROR → /autoware/system/duplicated_node_checker ERROR → AUTONOMOUS 차단. - rviz만 kill하면 잠시 후 다시 살아남.

    원인 1. rviz는 launch가 respawn한다. 런치 프로세스를 죽이지 않으면 계속 되살아난다. 2. 이전 스택의 오펀들(relay, image_transport/republish, robot_state_publisher, component_container)은 경로가 /opt/ros/humble/lib/...이라 /home/siruu/autoware/install 패턴 kill에 안 걸려 살아남는다.

    해결 - 런치 프로세스부터 종료 → 그 다음 자식/오펀 정리. - 오펀은 시작시각으로 구분해서 이전 세트만 제거: 

    ps -eo pid,lstart,cmd | grep -E "<pattern>" | grep "<이전 시각>" | awk '{print $1}' | xargs -r kill -9
    - 확인 시 pgrep -f "rviz2/rviz2"는 자기 명령까지 세므로 신뢰하지 말 것. ps -eo comm | awk '$1=="rviz2"' 로 확인한다.

    4. 라우팅 실패 — "Failed to find a proper route" / route is empty

    증상 SetRoutePoints가 실패하거나 route가 비어서 궤적이 안 생김.

    원인 CARLA autoware-contents Town10HD 지도의 lanelet에 Autoware가 요구하는 location / speed_limit / one_way 속성이 없음(샘플맵에는 있음).

    해결 scripts/fix_carla_lanelet_map.py — subtype=road인 lanelet 168개에 location=urban, speed_limit=30, one_way=yes 부여(.bak 생성). 이후 SetRoutePoints 성공.

    참고: 지도의 format_version: null 경고는 라우팅 실패의 원인이 아니었다(경고만 남음).

    5. "Ego is out of route" — 궤적이 2점으로 붕괴

    증상 route 설정은 success=True인데 궤적이 2점뿐이고 차가 안 움직임. 로그: behavior_path_planner.planner_manager: Ego is out of route, no module is running.

    원인 (두 가지) 1. 목표 계산이 heading을 무시했다. 기하학적 최근접 lanelet을 시작점으로 잡으면 반대 차선/역방향 lanelet이 걸려, mission_planner는 route를 만들어도 behavior가 거부한다. 2. ego가 실제로 도로 밖에 있는 경우. (주행 실패로 차선을 이탈해 정지하면 이 상태가 된다. 가장 가까운 lanelet 폴리곤 경계에서 2.68m 바깥이었다.)

    해결 1. scripts/find_routable_goal_heading.py — ego yaw와 lanelet 진행방향의 cos(Δ) > 0.3인 것 중 최근접을 시작 lanelet으로 선택. 2. 도로 밖이면 어떤 목표도 소용없다 → 재스폰(스택 재시작). lanelet2.geometry.inside()로 포함 여부를 먼저 확인할 것.

    6. [핵심] AUTONOMOUS가 안 걸림 — 3중 게이트

    컨트롤러를 스왑하고 나면 AUTO 버튼이 비활성/실패한다. 원인이 겹쳐 있어 하나씩 풀어야 했다.

    6-1. control_state STALE (컨트롤러 unload 부작용)

    • 증상 /autoware/control/performance_monitoring/control_state STALE → /autoware/control ERROR.
    • 원인 이 진단의 소스가 바로 우리가 unload한 controller_node_exe다. (autoware_launch/config/system/diagnostics/control.yaml: node: controller_node_exe, name: control_state)
    • 해결 scripts/control_state_diag.py — diag_graph는 /diagnostics의 status.name 문자열로 매칭하므로, 어느 프로세스가 발행해도 된다. 우리 노드가 실제로 control_cmd를 내는 동안만 띄우므로 "컨트롤러 활성"은 사실이다.

    6-2. trajectory_validation ERROR 깜빡임 → MRM 긴급정지

    • 증상 AUTO가 됐다가 곧 MRM_OPERATING으로 빠지며 정지. diag가 available↔ERROR로 깜빡임. 정작 planning_validator의 개별 검사(curvature/interval/lateral_accel/…)는 전부 L0(OK).
    • 원인 우리 브리지가 최종 토픽에 직접 궤적을 주입해서, validator의 입력 토픽 (/planning/scenario_planning/velocity_smoother/trajectory)이 비어 있었다. → validator 진단이 주기적으로 stale → 그래프 ERROR → 페일세이프. (게다가 behavior path가 ~8Hz 불규칙(간격 std 0.12s)이라 지터가 그대로 전파됨.)
    • 해결 scripts/path_to_trajectory.py를 고쳤다:
    • validator 입력 토픽으로 발행 (/planning/trajectory는 validator가 내보내므로 중복발행 금지)
    • 수신 즉시 통과가 아니라 최신 궤적 캐시 + 20Hz 고정 재발행(stamp 갱신)

    6-3. vehicle_cmd_gate stop-hold — Motion=STOPPED, Accept Start 비활성

    • 증상 우리 노드는 acceleration=+3.0을 내는데 게이트 출력은 -1.5(is_defined_acceleration=false). 차가 안 움직이고 "Accept Start"가 비활성.
    • 원인 우리 노드가 longitudinal.velocity를 채우지 않아 0이었다. 게이트는 velocity로 주행 의사를 판단해 "정지 계획"으로 보고 붙잡는다.
    • 해결 nmpc_autoware_node가 목표속도를 cmd.longitudinal.velocity에 채운다.

    7. 곡선 속도 — 5km/h 기어가기 → 자체 코너 감속으로 전환

    증상 코너 진입에서 속도가 5km/h까지 떨어짐. 단독 CARLA ROS2에선 없던 현상.

    원인 우리 노드가 Autoware planning의 per-point 속도 프로파일을 그대로 목표로 썼다. Town10HD의 코너는 대부분 교차로라 behavior_velocity_planner가 (게다가 perception이 꺼져 안전 확인을 못 하니) 매우 보수적인 기어가기 속도를 넣는다. 단독 구성에선 이 계층이 없고 ControlCore 자체 코너 감속(a_lat_max)을 썼기 때문에 빠르게 돌 수 있었다.

    해결 속도 주도권을 우리 컨트롤러로 회수: - 중간 구간은 고정 순항속도(cruise_speed_mps) → 코너 감속은 ControlCore가 담당 - 목표 근처(goal_zone_m = 궤적 끝까지 남은 호길이 12m 이내)에서만 Autoware 속도를 따라 정지 - cruise_speed_mps<=0이면 종전처럼 궤적 속도를 그대로 따른다(하위호환)

    튜닝 결과: a_lat_max=3.0은 코너 ~20km/h로 빨라 경로 이탈, 1.5는 코너 ~10~18km/h로 안정 완주. 직선은 순항 ~27km/h.

    남은 이슈 코너 진입 순간 속도가 target floor(14.4km/h=4.0m/s) 아래인 ~2.5km/h까지 언더슛 후 회복(near-stop). 급감속 오버슛으로 보이며, 속도센서 노이즈도 겹쳐 판독이 불안정하다. → 추후 작업(§9).

    8. 성능 — sim 7.3Hz → 15.9Hz (카메라 5개 끄기)

    증상 sim이 실시간의 ~0.4배(7.3Hz), 카메라 3.7Hz라 화면이 끊김. 단독 CARLA ROS2에선 20Hz가 나왔다.

    원인 Autoware 연동은 sync 락스텝이라 가장 느린 단계에 전체가 묶인다. 그중 큰 몫이 carla_sensor_kit의 카메라 6개(1600x900)를 매 tick 렌더링하는 비용이다. (그 외 NDT 스캔매칭, 플래닝, RViz 렌더링)

    해결 모니터링에 실제로 쓰는 건 CAM_FRONT 하나뿐이므로 나머지 5개를 비활성화: autoware_carla_interface/config/sensor_mapping.yaml의 enabled_sensors에서 주석 처리.

    before after
    sim clock 7.3Hz 15.9Hz
    CAM_FRONT 3.7Hz 8.0Hz

    주의: 이 파일은 install/...이 src/...로 심링크라 소스만 고치면 된다.

    9. 그 외 함정 (짧게)

    • CARLA sync 세션 중 제3 클라이언트 접속 금지. 디버깅하려고 별도 CARLA 파이썬 클라이언트를 붙이면 세션이 깨지고 ego가 소멸한다(Invalid session: no stream available with id). 디버깅은 ROS 토픽으로만 한다. 같은 이유로 carla_ros_bridge(→ /carla/ego_vehicle/...)를 autoware_carla_interface와 동시에 띄우면 안 된다.
    • VRAM 8GB: CARLA가 ~4GB를 쓰므로 perception:=true는 OOM(cudaErrorMemoryAllocation). 그래서 perception:=false + dummy_perception.py. 이 때문에 RViz RecognitionResultOnImage 패널이 'No Image' 가 된다 → scripts/cam_to_panel.py로 CAM_FRONT를 그 패널 토픽에 republish해서 채운다.
    • 속도센서 노이즈: 정지 상태에서 /vehicle/status/velocity_status가 ±30km/h까지 튄다. engage 시 "주행 중"으로 오인돼 실패하기도 한다(→ stop→autonomous 재사이클로 통과).
    • 재경로: 완주 후 새 목표는 clear_route → set_route_points. (route is already set 응답이면 clear를 안 한 것.)
    • 도착 위치: 목표에 도착해 멈춘 지점이 차선 경계에 걸리는 경우가 있어, 거기서 새 route를 잡으면 §5의 out-of-route가 난다.

    10. 추후 작업

    1. 속도센서 노이즈 개선 — NMPC 폴더의 SparseMHE 또는 EKF.hpp로 속도 추정 필터링.
    2. 코너 진입 오버슛(near-stop) 감소 — 코너 브레이크 감속률(control_core.hpp의 a_brake) 완화 등으로 급감속 언더슛 제거.
    3. Jetson Nano(nmpc) ↔ Desktop(Autoware/CARLA) 연동 — DDS로 분리 구동. 과거 docker0 인터페이스 블랙홀로 0Hz였던 이력 있음 → Fast-DDS interfaceWhiteList 필요.
    4. 타이어 모델 적용 — 대회 전까지는 타이어 동정 없이 대표적인 MF5.2 계수를 넣고, 그것으로 LUT를 계산해 Autoware에 적용.

    단위 주의: control_core.hpp의 속도 상수는 m/s다. max(4.0, ...)은 14.4km/h, max(5.0, ...)은 18km/h이지 4/5km/h가 아니다.
