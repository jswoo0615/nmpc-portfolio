    2026-07-20 — 계측 도입과 코너 near-stop 근본원인 규명

    전날 Autoware 연동 완주 후, 코너에서 속도가 크게 떨어지고 그 뒤 조향이 흔들리는 문제를 쫓았다. 결론부터: 가설 두 개가 데이터로 기각되고, 세 번째(경로 상실)가 진짜 원인이었다. 그리고 그 판정이 가능했던 것은 이날 만든 계측 도구 덕분이다.

    0. 왜 계측부터 만들었나

    좌표를 눈으로 훑으며 "나아진 것 같다"로 판단하다가 실제로는 악화된 변경을 통과시킬 뻔했다. scripts/drive_metrics.py — 주행 중 CTE / 속도 / 조향을 기록해 요약과 CSV를 남긴다. 동일 경로에서 변경 전후를 같은 지표로 비교하기 위한 것이며, 베이스라인 vs NMPC 비교 실험에도 쓴다.

    계측 자체가 두 번 우리를 속였다 (둘 다 도구를 고쳐 해결)

    1. 정지 표본 오염 — 표본 2155개 중 1286개가 정지 상태였다. 정지 중 조향 흔들림까지 세어 "지그재그 9.88회/초"가 나왔으나, 주행 표본만 보면 6~8회/초였다. → 조향·지그재그 통계는 주행 표본(v>0.3 m/s)만 사용하도록 수정.
    2. 도착 아티팩트 — "CTE 최대 2.895m"는 추종 실패가 아니었다. 목표 도착 후 궤적이 목표 스텁으로 붕괴해 "목표까지의 거리"를 CTE로 잰 것이다. (위치 (-102.1,-93.0), 목표 (-101.25,-95.8) → 거리 2.93m ≈ CTE 2.895m로 일치.) → 실제 주행 중 최악 CTE는 0.88m(코너 구간)이다.

    교훈: 지표는 구간을 나눠 봐야 한다. 하나의 요약 숫자는 쉽게 오해를 만든다.

    1. 베이스라인 (검증된 구성)

    lf 1.085 / lr 1.705, a_lat_max 1.5, cruise 8.33 m/s(30km/h), goal_zone 12m, dt 0.05

    속도 구간 표본 CTE 평균 CTE 최대 |dδ/dt| 평균
    고속 (>22 km/h) 288 0.090 m 0.291 m 0.094 rad/s
    중속 (14–22) 301 0.157 m 1.242 m 0.247
    저속 (7–14) 183 0.491 m 2.895 m 0.346
    초저속 (0.5–7) 96 0.227 m 2.878 m 0.083

    고속이 저속보다 5배 정확하다. "저속이 이런데 고속은 어떻겠나"는 걱정과 정반대이며, 고속은 회랑 연속화로 해결됨 / 저속 미해결이라는 기존 기록이 수치로 확인됐다.

    2. 기각된 가설 ① — 차량 기하 불일치

    가설: CARLA는 Tesla Model 3를 스폰하고 식별값은 lf 1.168 / lr 1.836(휠베이스 3.004m)인데, 런치는 Autoware sample_vehicle에 맞춘 1.085/1.705(2.790m)를 넘긴다. 모델이 7% 짧으니 r ≈ v·δ/L 상 같은 조향에 더 많이 돈다고 예측 → 조향을 덜 넣어 코너에서 밀릴 것이다.

    결과 (동일 스폰·동일 경로 A/B): 전 속도구간 악화 → 되돌림.

    구간 BASE CTE 평균 A1 CTE 평균
    고속 0.090 m 0.350 m
    중속 0.157 m 0.418 m
    저속 0.491 m 0.520 m

    |dδ/dt| 평균도 0.129 → 0.808 rad/s (6.3배).

    왜 틀렸나: 정적 축하중 비율(lr/L = 0.611)은 양쪽이 같지만 모멘트 암이 커져 슬립각 α_f = atan((v_y + l_f·r)/v_x) − δ 가 달라지고 타이어 작동점이 이동한다(하중이동). 그런데 Q 가중치·steer_slew·회랑 폭은 전부 lf=1.085 기준으로 스윕된 값이라 루프 게인 균형이 깨진다. → 기하를 실제에 맞추는 것 자체는 옳지만 재튜닝을 동반해야 한다. 단독 변경 금지.

    3. 기각된 가설 ② — 코너 브레이크가 과하다

    가설: 코너 진입에서 급감속해 언더슛한다 → a_brake(control_core.hpp)를 완화하자.

    결과: CSV 코너 구간(0.5초 간격)을 보니 점진적 언더슛이 아니었다.

    25.0s  17.1 km/h   steer -0.265
25.5s   0.0 km/h   steer -0.053   ← 0.5초에 17→0 (감속도 ~9.5 m/s²), 조향이 거의 0
26.0s   0.0 km/h
26.6s   1.5 km/h   (이후 천천히 회복 / 5km/h 미만이 2.5초 지속)

    a_brake를 아무리 튜닝해도 고쳐지지 않았을 문제다.

    4. ✅ 진짜 원인 — 경로 상실에 의한 비상정지

    조향 ≈ 0 + 급제동 조합은 nmpc_autoware_node의 경로 상실 분기와 정확히 일치한다.

    if (!odom_ok_ || !core_->has_path()) {
    cmd.lateral.steering_tire_angle = 0.0f;   // 조향 0
    cmd.longitudinal.acceleration  = -3.0f;   // 급제동
}

    그리고 has_path()가 왜 false가 되는지는 control_core.hpp의 set_path에 있다.

    if (m > 0 && hypot(xs[i]-px[m-1], ys[i]-py[m-1]) < 0.3) continue;  // 0.3m 중복 제거
if (m < 5) { spline_valid_ = false; return false; }                // 5점 미만이면 경로 무효

    has_path()는 spline_valid_를 그대로 돌려준다. 즉 중복 제거 후 5점 미만이면 직전까지 멀쩡했던 경로를 즉시 버린다. 실제로 이날 궤적이 2~3점으로 붕괴하는 것을 여러 번 관측했다(out-of-route 시 2점, 도착 후 3점/1.0m).

    인과 사슬

    교차로에서 궤적이 순간적으로 짧아짐
  → 0.3m 중복 제거 후 5점 미만
  → set_path 실패, spline_valid_ = false
  → has_path() == false
  → steer 0 + accel −3.0 (급정지: 17→0 km/h in 0.5s)
  → 정지 후 저속 모델 불일치(max(1.5, vx), max(5.0, ..))로 회복 지연 2.5초
  → 그 대역에서 CTE 0.88m로 악화

    고칠 방향

    1. 처리 쪽 (효과 확실, 작업량 작음) — 궤적이 잠깐 나빠도 직전 유효 스플라인을 짧은 유예시간(예: 0.3~0.5s) 동안 유지하고, 유예를 넘겨야 감속한다. Control/SafeBuffer.hpp(마지막 유효 궤적 + generate_fallback_control)가 이미 있는데 이 경로에서 쓰이지 않고 곧장 급정지로 간다. 인프라는 이미 갖춰져 있다.
    2. 입력 쪽 — 왜 교차로에서 궤적이 짧아지는지(behavior planner의 교차로 처리인지, lanelet 전환 순간의 공백인지) 확인.

    급정지가 사라지면 저속 대역(7–14 km/h)에 들어가는 일 자체가 줄어드므로, 저속 조향 지그재그도 함께 완화될 가능성이 크다. 두 문제는 사실상 하나로 다뤄야 한다.

    5. 이날의 방법론적 교훈

    • 한 번에 하나만 바꾼다. 차량 기하와 브리지 구조를 동시에 바꿔 원인 분리에 실패했고, 주행이 악화된 상태로 되돌려야 했다.
    • 눈이 아니라 지표로 판정한다. 다만 지표도 구간을 나눠 봐야 하며, 정지·도착 구간은 분리하지 않으면 결론을 왜곡한다.
    • 그럴듯한 가설일수록 측정으로 죽인다. 기하 불일치도, 코너 감속 과다도 물리적으로 타당해 보였지만 둘 다 틀렸다.
