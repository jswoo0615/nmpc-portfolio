    2026-07-18 세션 트러블슈팅 전체 기록

    CARLA 0.9.15 + NMPC 자율주행 스택의 하루 디버깅 세션에서 발견·수정한 모든 문제의 기록. 증상 → 근본 원인 → 수정 → 검증 순으로 정리. 모든 수정은 맵 비종속 (Q 가중치 등 튜닝값의 맵별 변경 금지 제약 준수).

    최종 성과 요약

    지표 세션 시작 세션 종료
    기준 경로(spawn91→16) 최대 횡오차 2.93 m 0.82 m (−72%)
    평균 횡오차 (frenet_d) — 0.160 m
    코너 통과 속도 9~13 km/h 17~25 km/h (인간 자유류 수준)
    평균 속도 27 km/h 36 km/h
    멀티맵 — Town10HD/Town03/Town05 무수정 완주

    A. 빌드/배포 문제

    A1. make -j4 실패 — OSQP API 불일치

    • 증상: test_solver_benchmark.cpp 컴파일 에러.
    • 원인: 코드는 OSQP v1.x API, 시스템(ROS Humble)은 0.6.2.
    • 수정: __has_include(<osqp_api_constants.h>) 기반 듀얼 API 가드. 0.6.x 경로는 csc_matrix/OSQPData/OSQPWorkspace + c_int→OSQPInt typedef.
    • 파일: test/test_solver_benchmark.cpp

    A2. 구버전 .so — nmpc_core has no EKFWrapper

    • 원인: examples 폴더의 .so가 빌드 산출물과 불일치.
    • 수정: 빌드 후 nmpc_core.so와 nmpc_core.cpython-310-x86_64-linux-gnu.so 양쪽 모두 복사하는 절차 확립.

    A3. Python 환경

    • conda python3.13은 cpython-310 .so와 비호환. 시스템 파이썬 /usr/bin/python3 (3.10.12) 고정 (carla 모듈도 여기에만 있음).

    B. 초기 실행 불능 문제

    B1. NameError: yaw_rad (planner)

    • 원인: 이전 커밋이 정의만 지우고 사용처를 남김.
    • 수정: yaw_rad = math.radians(transform.rotation.yaw) 복원.

    B2. 직진도 못 함 (Acc −0.75 상시 제동) — IPM 발산 kkt~1e249

    • 원인 사슬 (3중):
    • execute_fallback이 값 반환 방식이라 결과가 무시됨 → 참조 전달로 변경.
    • 위반된 d-회랑(±0.8m 밖, 실측 d 최대 8.4m)에서 프라이멀-듀얼 배리어의 λ가 무한 성장 → P 행렬 8e5→2.6e10, Newton step 2280. → update_dual에 슬랙/멀티플라이어 클램프 (s ≥ 1e-6, λ ∈ [1e-10, 1e4]).
    • 차가 회랑 밖에 있으면 솔브 전 회랑을 현재 d+1.0m로 일시 확장(적응 회랑), 솔브 후 복원.
    • 파일: include/nmpc/Control/SparseNMPC_IPM.hpp, src/nmpc_wrapper.cpp
    • 검증: 결정론적 리플레이로 재현 후 소멸 확인.

    B3. C++ cold_restart로 부족한 발산 복구

    • 증상: 상태 리셋 후에도 동일 입력에서 재발산 (투영/듀얼 상태 누수).
    • 수정: Python 레벨에서 kkt>1e6 또는 NaN 시 솔버 객체 재생성 + 해당 틱 코스트(steer/accel=0). 8/8 복구 검증.
    • 파일: nmpc_core_planner.py (run_step 발산 복구 블록)

    B4. mu(노면 마찰 추정)가 0.1~0.3으로 붕괴 → "빙판 모드" 상시 제동

    • 원인: EKF의 vy가 발산(vy/mu 비가관측), 코너에서 특히 악화. |ay| 게이팅 1차 시도는 코너에서 오히려 악화.
    • 수정: 솔버에는 mu=1.0 고정 공급, EKF raw 값은 CSV 로깅만. (EKF vy 가관측성 확보는 향후 연구 과제)

    B5. kappa_ref 부호 반전으로 코너링 피드포워드 상쇄

    • 원인: config.kappa_ref[k] = -raw_kappa — 투영(d/mu 측정)과 예측 모델(mu_dot=r−κ·s_dot)의 프레임 불일치.
    • 수정: 부호 반전 제거(스플라인 원 부호 유지). 코너 정상 조향 −0.14→−0.188 회복.
    • 파일: src/nmpc_wrapper.cpp

    C. 차량 동정 (identify_vehicle.py 신규 작성)

    C1. 모델-실차 파라미터 불일치 (코너 언더스티어의 근본 원인 1)

    • 실측 (CARLA Tesla Model 3): m=1845kg (모델 1500), wheelbase=3.005 (2.8), lf=1.168/lr=1.836 (1.2/1.6), max_steer 70°=1.222rad, 조향 맵핑 저각 기울기 ≈1.15 (모델 1.22).
    • 코너링 강성: Cf=90.1k, Cr=54.5k N/rad → c1_front 8.15, c1_rear 7.74 — 모델의 c1=15.0이 실측의 약 2배 (모델이 타이어 힘을 2배로 믿어 조향을 절반만 명령).
    • 반영: RealTimeDynamicsModel6DOF.hpp/RealTimeDynamicsModel.hpp/HighFidelityDynamicsModel.hpp의 veh_params, wrapper lf/lr, planner max_steer_physics 1.22→1.15, tire c1.
    • 동정 스크립트의 자체 버그들:
    • 수동 1단 고정이 목표 속도 도달을 막음 → 자동변속.
    • set_transform 텔레포트가 이전 속도 벡터 유지 → set_target_velocity(0)/set_target_angular_velocity(0) 명시 리셋.
    • P 제어기 정상상태 오차로 도달 판정 실패 → 게인/허용치 조정 (피팅은 실측 vx 사용이므로 무해).
    • 슬립각 부호 오류: local_vel의 vy(CARLA 우+)를 한 번 더 뒤집어 vy만 ISO, r/δ는 CARLA인 혼합 프레임 → 슬립각 4~5배 과대, C 과소. CARLA 좌수계는 ISO의 거울상이므로 전부 CARLA 부호로 통일.

    D. 코너 동결 (핵심 문제)

    D1. 동심원 호 가짜 평형 — 조향 −0.145 (raw −0.167) 3.5초 고정, cte 2.5m

    • 증상: 코너에서 조향이 상수로 얼어붙고 차는 참조와 평행한 동심원 호(kappa 0.055 vs 도로 0.061, 정확히 1/(R+d)) 주행.
    • 진단 과정:
    • raw_steer 계측 추가 → 후처리(EMA/정규화) 무죄, 솔버 원출력 자체가 동결.
    • 콜드 스타트 리플레이 → 같은 상태에서 즉시 −0.24~−0.28 출력. OCP 최적해는 정상, 웜스타트가 국소최적점에 갇힘.
    • 기전: Q_kappa_track 비용(r≈κ·s_dot)은 s_dot이 반경비로 스케일되어 모든 동심원 호에서 정확히 0 → "평행 호 계곡"이 비볼록 문제의 국소최적점.
    • Frenet 분모(1−dκ)/부호 정합은 수치 검증으로 무혐의 확인.
    • 수정 반복:
    • Python 1회성 콜드 리스타트 → 수십 틱 만에 재수렴 (부족).
    • Python 콜드 모드(매 틱 재생성) → 8회 반복 항상 Exhausted, 경계 밖 쓰레기 출력(raw −2.08) + 투영 연속성 상실 → 10틱마다 재생성 + 4초 타임아웃 + 출력 클램프(steer ±0.6, accel [−10,3]).
    • 최종: C++ reset_warm_start() — wrapper에서 |frenet.d| > 0.7이면 U_guess/듀얼/배리어만 매 솔브 리셋, u_last(레이트 앵커)와 투영 상태는 유지. 트리거 지연 0. Python 콜드 모드는 cte>3.0 비상용으로 강등.
    • 파일: SparseNMPC_IPM.hpp(reset_warm_start), nmpc_wrapper.cpp, nmpc_core_planner.py

    D2. reference_generator의 역효과 로직 2건

    • 차선변경 완화: cte>1.2에서 q_d −40%/q_kappa_track −80% — 참조 점프와 차량 이탈을 구분 못 해 이탈할수록 복귀를 포기하는 양성 피드백 → 제거. (곡률 항만 cte 0.8~1.5에서 페이드하는 형태로 재도입 — q_d는 절대 유지)
    • 저속 조향 억제: vx<5.0m/s(=18km/h!)에서 조향 페널티 급증 — 코너링 속도 영역 침범 → 임계 1.5m/s로 축소.

    E. 속도 문제 ("교차로에서 9km/h는 비인간적")

    E1. 코너 재가속 봉쇄

    • 원인: cte>0.5m이면 d_blend가 포화되어 R_AccelRate +1000 → 코너 내내 12~15km/h 갇힘. 목표 하한(18km/h)이 아니라 이 커플링이 주범.
    • 수정: 커플링 +300, 급코너 목표 하한 18→24km/h (인간 자유류 20~30km/h, ~0.25g 근거. R=16.3m에서 24km/h는 슬립각 ~2°로 선형 영역).
    • 결과: 코너 통과 18~25km/h, 추종 오차는 오히려 개선.

    F. 경로/참조 전처리 문제 (무작위·차선변경 경로)

    F1. GRP(GlobalRoutePlanner) 경로 품질

    • 중복 웨이포인트(0거리 세그먼트 → 스플라인 병리) → dedupe.
    • 차선변경이 10.6m 대각선 한 세그먼트로 출력 → S-블렌드(20m 코사인 램프)로 변환.
    • 검출 강건화 반복: (i) 이전 방향 기준 → 경로 맨 앞 점프 놓침, (ii) 고정 인덱스 오프셋 기준 방향 → 대각선 주변 오발동, (iii) 최종: 세그먼트 길이 게이트(중앙값 2.2배↑) + 착지 이후 세그먼트 방향 기준 횡오프셋 >1m.

    F2. 차선변경 이륙점 pop → 참조 순간이동

    • 증상: 버퍼 시작이 차보다 10m 앞으로 점프, frenet_s=0 포화 + fd −9m 쓰레기 (스플라인 정의역 밖 투영).
    • 수정 시도 중 사고 2건: 에고 기준 접착(참조를 차 위치로 굽힘)이 fd=0인데 실제 6m 이탈을 가리는 위험을 두 번 재생산 → 주행 중 접착 절대 금지 원칙 확립.
    • 최종 설계:
    • 종방향 커버: 참조 자체 방향 후방 외삽 (횡오프셋 보존 → 가림 원천 불가).
    • 접착 램프는 정지 상태(스폰, vx<2)에서만.
    • 앵커는 콜드모드/OFF_ROUTE 억제 플래그로만 사용. 예산: 주행거리 40m 또는 시간 1500틱 (거리 전용 예산은 정지 차량에서 영구 미만료 → 데드락 사고, G3 참조).
    • 합류 실행은 솔버의 웜리셋 복구 기제가 담당 (fd가 참값 3.5m를 보고하므로).

    F3. 코너 컷 pruning

    • 원인: 전방 40°+ 코너 감지 시 pass_dist를 5m로 넓혀 꼭지점 웨이포인트를 미리 pop → 스플라인 시작 기하가 코너 중간에 요동 (fd 0.25초에 1.8m 점프).
    • 수정: 제거. "차 뒤로 지나감(dot<0, 8m 이내)" 규칙만으로 충분.

    F4. 로터리(Town03)에서 pruning fallback 점프

    • 원인: pop=0일 때 "최근접 wp까지 pop" fallback이 원호 건너편(직선거리로 더 가까움) wp로 점프 → cte 0.3→7.0 한 틱.
    • 수정: fallback은 버퍼 머리가 8m 이상 뒤처졌을 때만 작동.

    F5. 자기교차 경로에서 투영 가지 잠금

    • 원인: 727m 경로가 같은 교차로를 s=190/s=410에서 재통과 — 400m 스플라인 창에 두 가지가 다 들어와 투영이 반대 가지에 잠김(조향 부호 반전, 이탈 중 가속).
    • 수정: solver_wp_limit=80 (≈160m; 5초 호라이즌 최대 요구 ~85m의 2배 여유).

    G. 안전장치 상호작용 사고

    G1. Python 콜드 모드 매 틱 재생성 → 쓰레기 출력으로 차량 웨지 (D1-2 참조)

    G2. runner OFF_ROUTE 가드 vs 스폰 합류

    • 원인: GRP가 시작점을 9.4m 옆 도로에 스냅(Town03 spawn235) → 합류 중 cte>6으로 즉사.
    • 수정: lc_flag 활성 중 가드 유예 — 단 절대 상한 cte 12m 병행.

    G3. 데드락 (사용자 발견: "시뮬레이션이 제대로 안 됨")

    • 원인: G2 유예 + F2 예산(주행거리 전용)의 합작 — 정지 차량은 예산이 만료되지 않아 lc_flag 영구 유지 → OFF_ROUTE 영구 유예 → 110초+ 공회전.
    • 수정: 앵커 시간 만료(1500틱) + cte 12m 절대 상한. 병적 경로는 이제 4.2초에 정상 종료.

    H. 추종 오차 추가 축소 (동정 모델 기준 재튜닝 — 전역 값)

    기존 값들은 타이어 강성 2배 과대 모델 기준의 보수 튜닝. 동정 모델은 실효 루프 게인이 절반이므로 동일 안정 여유에서 조임 가능: - wrapper straight_gain 하한 0.075 → 0.25 (직선 0.3m 표류 제거) - 웜리셋 임계 1.0 → 0.7m (코너 진입 동결 조기 차단) - q_d_base 300 → 500 - 결과: 91→16 fd 최대 1.06→0.82m, 평균 0.198→0.160m, fd>1m 0%, 피시테일 없음(조향 부호반전 3회/37초). 25→91 평균 0.660→0.536, 105→84 0.399→0.340 — 회귀 없음.

    부록: JetsonNano 병합 감사 (같은 날 후속)

    • JetsonNano 브랜치(CUDA/Jetson 배포 인프라)를 dev1에 병합 — 충돌 0, x86 빌드/주행 수치 동등성 실증 (91→16 fd 0.93/0.146).
    • 수학 라이브러리 diff는 CUDA_CALLABLE 주석 + 호스트/디바이스 분기(호스트 경로 std:: 동일)로 무해.
    • 잠재 함정: StaticMatrix initializer_list 생성자의 해석이 저장소 순서(column-major)에서 row-major 재해석으로 변경됨. 현재 2D 행렬 브레이스 초기화 사용처는 없지만, 향후 2D 행렬을 중괄호로 초기화할 때 구버전 습관(열 우선 나열)으로 쓰면 조용히 전치됨. 벡터(StaticVector)는 순서 불변.

    부록 2: MF52 타이어 업그레이드 (같은 날 후속)

    • 발견: 6DOF 예측 모델의 횡력은 100% MF52 LUT 경로(tire_lut.getForces)에서 나오며, 이전에 동정 반영한 tire_params_base.c1은 이 경로에서 미사용(무효)이었음. 기본 MF52 파라미터의 강성은 실측 대비 전륜 +26%/후륜 +116% 과대.
    • 비선형 동정 (identify_tire_mf52.py): 스텝 조향 과도 중 힘/모멘트 평형 2식(m·ay, Iz·ṙ)으로 매 샘플 축력 역산 — 정상원 선회 없이 α 0~10° 커버(1520 샘플). 선형 강성이 정상상태 동정과 교차 일치(전 91.4k vs 90.1k N/rad).
    • CARLA 타이어의 실체: PhysX tire_friction=3.5 — 도로 주행 영역에서 사실상 포화하지 않는 선형 타이어. 포화 형상(C/E)은 데이터로 동정 불가 → 표준값(C=1.4, E=−0.5), PDY1=3.5.
    • LUT 구조 함정: 런타임 getForces는 FNOMIN에서 만든 정규화 곡선을 fz에 선형 스케일(×load_modifier) — PKY2의 하중 의존 Kya는 반영 안 됨. 따라서 공칭 기울기 k0를 양축 절충(8.477/rad)으로 설정: 유효 강성 전륜 −3.3%/후륜 +3.5%.
    • 통합: carla_tesla.tir 생성(PKY1=9.183, PKY2=1.5, 시프트/캠버/복합슬립 항 0), wrapper load_tire_model() 추가, planner 솔버 생성 3지점을 _make_solver()로 통합(재생성 시 타이어 유지).
    • 검증: 91→16 fd max 0.82→0.67m/avg 0.152, 36.9km/h, Fast Exit 2434→2831·Exhausted 1 (모델 정합 → 웜스타트 품질 선순환). 105→84 avg fd 0.340→0.220(−35%), 29.6→31.9km/h.

    부록 3: .tir 일반화 + 조향 지그재그 + 멀티맵 재검증 (같은 날 후속)

    .tir 자동 캘리브레이션 (임의 표준 타이어 파일 지원)

    • 요구: CARLA 역산 .tir뿐 아니라 임의의 표준 .tir 계수로부터 마찰/강성을 역산해 제어 가능해야 함.
    • 1차 구현(mu_0 단일 스케일): 기울기를 맞추면 피크가 0.64g로 왜곡 → 하드 경로 저하 확인.
    • 최종: MF52 정식 스케일 채널 분리 보정 — 로드 시 .tir 자체 계수에서 k0(공칭 정규화 강성)과 피크 mu를 프로브하고, LKY = k0_동정/k0_tir(부호 포함 — ISO 관례의 음수 Kya 자동 흡수), LMUY = mu_동정/mu_tir로 독립 정합. wrapper load_tire_model(path, lky, lmuy), planner _compute_tire_scales().
    • 검증: representative_mf52.tir(LKY=−0.635, LMUY=3.057)로 기준 경로 fd 0.63/0.151 — carla_tesla와 동등.

    조향 지그재그(펌핑) 조사 — 부분 해결, 대회 전 필수 완결 과제

    • 증상(사용자 관찰): 곡선에서 조향을 ±0.2rad(휠 ~11.5°) 줬다 풀었다 반복.
    • 배제된 원인: 웜리셋 매 솔브 발동(0.5초 쿨다운 부여 — 지표 불변으로 주범 아님이 판명, 단 콜드 분산 억제 차원에서 유지).
    • 수정된 원인: 적응 회랑의 자기참조 토글 — |d|가 0.6 문턱을 오갈 때 배리어 경계가 0.8↔d+1.0으로 매 틱 점프 → QP 해가 함께 점프. 확장은 즉시, 수축은 틱당 1cm 연속 감쇠로 배리어 연속화.
    • 효과: 고속(>50km/h) 구간 지그재그 전 주행 0~5회 (Town04 고속도로 진입 40m 전 흔들림 해소, entry 740m 경로 fd 평균 0.099).
    • 저속 구간에 지표상 남는 반전은 틱 검증 결과 대부분 실제 참조 기하(램프 치케인/차선변경 S-블렌드) 추종 — 단 저속 실제 지그재그가 완전히 0은 아님. 대회 전 반드시 완결해야 하는 필수 과제 (사용자 지정). 후보 원인: Suboptimal 솔브의 해 분산, kappa-fade/straight_gain 게인 전환, EMA α=0.5 통과 대역.

    Town04 고속도로 + 전맵 재검증 (모두 무수정)

    • Town04 진입 램프 ×2, 출구 램프 ×1: 3/3 도착. 고속도로 추종 fd 평균 0.099m.
    • 전맵 배치 8/8 도착: Town03 로터리(순간 과도 7.7 복구), Town05 ×2, Town02(첫 시도 맵) ×2 — 무수정 fd 평균 0.14~0.17m, Town10 기준 3경로(91→16 fd max 0.65 최고 기록).
    • 남은 과도 이슈(복구됨, 문서화): 로터리 투영 순간 과도, 다차선 점프 참조 스냅(fd 5).

    부록 4: ROS2 노드화 (같은 날 후속)
